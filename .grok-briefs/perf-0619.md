## Optimization (verified low-risk, byte-identical output)
File: `src/cluxion_runtime/core/preprocess.py:173` `estimate_tokens()`.
Current body counts non-ASCII code points with a per-character Python generator: `cjk = sum(1 for ch in text if ord(ch) > 127)`, then returns `max(1, cjk + (len(text) - cjk) // 4)` (ASCII weighted 1/4). The Python-level per-char loop is the bottleneck.
Fix: compute the same cjk count at the C layer. `ascii_chars = len(text.encode('ascii', 'ignore')); cjk = len(text) - ascii_chars`. ASCII-encodable code points are EXACTLY those with `ord <= 127`, so `len(text) - ascii_chars` reproduces `sum(1 for ch if ord>127)` bit-for-bit. Keep the rest of the formula IDENTICAL (the `max(1, ...)`, the `// 4` ASCII weighting, same return).
This is the Python mirror of Rust `context.rs::estimate_tokens` (parity-tested). KEEP the Rust side unchanged — the result is identical so Stage-1 parity holds.
Measured: 256k-char message 3.48ms→0.38ms (9.1x); full `compress()` on a realistic 183k-token/80-msg conversation 21.3ms→2.2ms (9.6x), byte-identical tokens_before/tokens_after/stages_applied.
